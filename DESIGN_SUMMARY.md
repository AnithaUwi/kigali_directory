# Design Summary: Kigali Services Directory

## Database Architecture

### Firestore Collections

#### 1. **Users Collection** (`users/`)
Purpose: Store user profile information and metadata

**Schema:**
```
users/{userId}
├── uid (string) - Firebase Auth UID
├── email (string) - User's email address
├── displayName (string) - User's chosen display name
├── emailVerified (boolean) - Email verification status
└── createdAt (timestamp) - Account creation time
```

**Example Document:**
```json
{
  "uid": "abc123xyz",
  "email": "john.doe@example.com",
  "displayName": "John Doe",
  "emailVerified": true,
  "createdAt": "2024-03-01T10:30:00Z"
}
```

**Design Rationale:**
- UID as document ID ensures one profile per authentication user
- Email stored for quick lookups if needed
- Timestamp enables sorting users by registration date
- Separate from Firestore Auth system for flexibility

#### 2. **Listings Collection** (`listings/`)
Purpose: Store all service and place listings in Kigali

**Schema:**
```
listings/{listingId}
├── title (string) - Service/place name
├── description (string) - Detailed description
├── category (string) - Service category
├── latitude (number) - Geographic latitude
├── longitude (number) - Geographic longitude
├── address (string) - Physical address
├── phone (string) - Contact phone number
├── website (string) - Website URL
├── imageUrl (string) - Photo URL
├── createdBy (string) - User ID of creator
├── createdAt (timestamp) - Creation date
└── updatedAt (timestamp) - Last modification date
```

**Example Document:**
```json
{
  "title": "Kigali Central Hospital",
  "description": "Main public hospital in Kigali",
  "category": "Hospital",
  "latitude": -1.9441,
  "longitude": 29.8739,
  "address": "KN 3 Ave, Kigali",
  "phone": "+250-788-XXX-XXX",
  "website": "https://hospital.rw",
  "imageUrl": "gs://...",
  "createdBy": "user123",
  "createdAt": "2024-02-15T09:45:00Z",
  "updatedAt": "2024-03-05T14:20:00Z"
}
```

**Design Rationale:**
- Each field directly maps to app display/functionality
- Latitude/longitude enable map integration
- `createdBy` field enables ownership-based queries and security rules
- Timestamps enable sorting and audit trails
- Document ID auto-generated by Firestore for uniqueness

### Security Rules

**File:** `firestore.rules`

```
rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {
    
    // Users collection - readable by all, writable by self
    match /users/{userId} {
      allow read: if true;
      allow create: if request.auth != null;
      allow update, delete: if request.auth.uid == userId;
    }
    
    // Listings collection - readable by all
    // Create allowed for authenticated, Update/Delete only by owner
    match /listings/{listingId} {
      allow read: if true;
      allow create: if request.auth != null;
      allow update, delete: if request.auth.uid == resource.data.createdBy;
    }
  }
}
```

**Security Model:**
- **Public Read**: Anyone can browse listings and user profiles
- **Authenticated Create**: Only logged-in users can create listings
- **Ownership-Based Update/Delete**: Users can only edit/delete their own listings
- **Rule Verification**: `createdBy` field validates user ownership

**Trade-off:**
- Simple rules prioritize usability over privacy
- All users can see all listings (considered public data)
- Users' personal details only include public-facing profile info

**Design Decisions:**
- **Immutable fields**: All final (functional programming style prevents bugs)

## State Management Architecture

### Provider Pattern Implementation

**Layer 1: Services** (Direct Firebase Operations)
```
auth_service.dart
├── signUp(email, password, displayName)
├── signIn(email, password)
├── signOut()
└── resendVerificationEmail()

listing_service.dart
├── createListing(ListingModel)
├── getListingsByUser(userId)
├── updateListing(ListingModel)
└── deleteListing(listingId)
```

**Design:**
- Pure functions with no UI dependencies
- Testable and reusable
- Encapsulate Firebase API details

**Layer 2: Providers** (State Management)
```
auth_provider.dart
├── user: User? (current authenticated user)
├── isLoading: bool
├── errorMessage: String?
└── Stream<User?> authStateChanges()

listing_provider.dart
├── listings: List<ListingModel>
├── filteredListings: List<ListingModel>
├── searchQuery: String
├── selectedCategory: String?
└── initializeUserListingsStream(userId)
    initializeAllListingsStream()
    setSearchQuery(String)
    setCategory(String?)
```

**Design:**
- Consumer pattern for rebuilds (only necessary widgets rebuild)
- Separate streams for all vs. user listings
- Notifications on data changes
- Public getters, private setters

**Layer 3: UI** (Screens and Widgets)
```
create_listing_screen.dart
├── Consumes: ListingProvider, AuthProvider
├── Actions: Create, Save to Firestore
└── State: Form fields, validation

directory_screen.dart
├── Consumes: ListingProvider
├── Actions: Search, Filter, Navigate
└── State: Search text, selected category

my_listings_screen.dart
├── Consumes: ListingProvider, AuthProvider
├── Actions: Edit, Delete, Navigate
└── State: User's listings stream
```

**Design Benefits:**
```
Service Layer (Testable)
       ↓
Provider Layer (Reusable state)
       ↓
UI Layer (Clean, focused on display)
```

## API Integration Points

### Firebase Authentication
- **Signs in users** with email verification
- **Stream listening**: authStateChanges() auto-updates when user logs in/out
- **Profile data**: User object provides uid and email

### Cloud Firestore
- **Real-time streams**: Changes in database auto-update app UI
- **Transactions**: Create + profile creation atomic operation
- **Queries**: Filter by createdBy for user-specific listings
- **Security**: Rules enforce ownership at database level

### OpenStreetMap (flutter_map)
- **Tile Layer**: Maps download tiles from OSM servers (free)
- **Markers**: Colored circles show listing locations
- **Map Center**: User's current location via Geolocator
- **No API key required**: Eliminates cost and setup complexity

### Google Maps Navigation Intent
- **Launch navigation**: Intent opens Google Maps for directions
- **Benefits**: User has familiar navigation experience
- **No integration code**: Platform handles via deep linking

## Design Trade-offs & Decisions

### Trade-off 1: Client-Side vs. Server-Side Filtering
**Decision:** Client-side filtering (search, category filter)

**Pros:**
- Instant feedback to user
- No extra Firestore read operations
- Offline capability (if cached)

**Cons:**
- Filters entire dataset in memory
- Slower with 10,000+ listings
- Extra data transfer

**Rationale:** Kigali services directory expected to have <1000 listings; client-side filtering acceptable.

**Alternative for scale:** Server-side filtering via Firestore queries would be needed for millions of listings.

### Trade-off 2: Monolithic Providers vs. Multiple Small Providers
**Decision:** Three main providers (AuthProvider, ListingProvider, SettingsProvider)

**Rationale:**
- Monolithic: Single source of truth, easier testing
- Would split only if each provider served completely different domain
- Current split by feature (auth, listings, settings) is logical

### Trade-off 3: Timestamps in Firestore
**Decision:** Store both createdAt and updatedAt

**Rationale:**
- createdAt: Sort listings by newest
- updatedAt: Show "Last edited" info to users
- Both enable audit logging

### Trade-off 4: Category as String vs. Enum
**Decision:** Store as string in Firestore, convert to enum in app

**Rationale:**
- Strings are flexible (database can add categories without app update)
- Enum in app provides type safety and IDE autocomplete
- Conversion at boundaries is clean pattern

### Trade-off 5: Ownership Field (createdBy)
**Decision:** Store user UID in createdBy field

**Rationale:**
- Firestore security rules can verify ownership: `request.auth.uid == resource.data.createdBy`
- Enables finding "My Listings" with efficient query: `where('createdBy', isEqualTo: userId)`
- Allows "Listings by User" profile feature

## Technical Challenges & Solutions

### 1. Demo Mode Hiding Real Integration Issues
**Problem:** App appeared functional with demo data while real Firebase was broken

**Solution:** Feature flag (`DEMO_MODE`) clearly toggles between mock and real
- Disabled for production testing
- Enables rapid development feedback

### 2. Orphaned Authentication Accounts
**Problem:** Users deleted from Firestore could still log in

**Root Cause:** Auth and Database are separate systems

**Solution:** 
- Auto-recreate missing profiles on login
- Never just delete from database; delete from Auth Console
- User-facing soft deletions (mark inactive) safer than hard deletes

### 3. Form Labels Invisible on White Background
**Problem:** White form labels on white input field background

**Solution:** Changed TextStyle color to `Colors.black54` (87% opacity)
- Visible on white background
- Maintains visual hierarchy
- Tested for accessibility

### 4. Category Filter Not Scrollable on Mobile
**Problem:** Category chips wrapped and pushed content off screen

**Solution:** Wrapped in `SingleChildScrollView` with `Row`
```dart
SingleChildScrollView(
  scrollDirection: Axis.horizontal,
  child: Row(
    children: [/* FilterChip buttons */],
  ),
)
```

## Performance Considerations

### Database Queries
- **User Listings**: Filtered at Firestore level with `where('createdBy', isEqualTo: userId)`
- **All Listings**: Full download (expected <1000 documents)
- **Search/Filter**: Client-side filtering from downloaded data

### Stream Management
- **Two streams**: userListings and allListings
- **Conditional subscription**: User stream only active when needed
- **Automatic cleanup**: Provider disposes streams on unmount

### Image Storage
- **Google Cloud Storage**: Images stored in Firebase Storage bucket
- **Lazy loading**: Images only download when card displayed
- **No image processing**: Full-resolution images (acceptable for <1000 listings)
